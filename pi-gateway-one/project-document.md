# โครงการ Smart CoE Monitoring

ระบบ IoT สำหรับตรวจวัดและรวบรวมข้อมูลสภาพแวดล้อมจากห้องต่างๆ ภายในอาคารภาควิชาวิศวกรรมคอมพิวเตอร์ (CoE) ข้อมูลทั้งหมดจะถูกส่งขึ้นระบบ Cloud เพื่อนำไปแสดงผล วิเคราะห์พยากรณ์ล่วงหน้า และแจ้งเตือนเมื่อเกิดความผิดปกติแบบเรียลไทม์

---

## 1. อุปกรณ์ฮาร์ดแวร์และเซ็นเซอร์ (Hardware & Sensor Nodes)

แต่ละห้องจะถูกติดตั้ง **Multimodal Board** ทำหน้าที่เป็นโหนดสำหรับวัดค่าสภาพแวดล้อมต่างๆ และส่งข้อมูลไปยัง Sensor Gateway โดยบอร์ดแต่ละตัวสามารถเลือกประกอบเซ็นเซอร์ได้ตามความเหมาะสมของแต่ละพื้นที่ใช้งาน ซึ่งรองรับเซ็นเซอร์ดังต่อไปนี้:

| เซ็นเซอร์ | หน้าที่ |
|---|---|
| **DHT22** | วัดอุณหภูมิและความชื้น |
| **IR Sensor** | อินฟราเรดตรวจจับวัตถุ |
| **LDR Sensor** | วัดความสว่างของแสง |
| **MQ-2** | ตรวจจับก๊าซไวไฟและควัน |
| **Sound Sensor** | ตรวจจับเสียง |
| **Hall Effect Sensor** | ตรวจจับสนามแม่เหล็ก |

## 2. โครงสร้างเครือข่ายและเส้นทางการส่งข้อมูล (Network Topology by Floor)

การส่งข้อมูลถูกออกแบบให้เหมาะสมกับข้อจำกัดและลักษณะของแต่ละชั้นในอาคาร ดังนี้:

### ชั้น 2

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง R201, Co-Ai และ R200 พร้อมทั้งติดตั้งกล้อง ESP-CAM ที่ห้อง Co-CoE
- **การส่งข้อมูล**: อุปกรณ์ทั้งหมดจะส่งข้อมูลผ่านโปรโตคอล **ESP NOW** มารวมกันที่ Sensor Gateway (ตั้งอยู่ที่ห้อง Co-CoE) จากนั้นจะส่งต่อไปยัง Network Gateway โดยเลือกใช้โปรโตคอลที่เหมาะสมตามระยะทาง เช่น BLE, ESP NOW หรือ RF Communication

### ชั้น 3

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง AI และห้อง Network
- **การส่งข้อมูล**: ข้อมูลของทั้งสองห้องจะถูกส่งผ่านโปรโตคอล **CoAP** ซึ่งมีน้ำหนักเบาและเหมาะสมอย่างยิ่งกับอุปกรณ์ที่มีทรัพยากรจำกัด โดยข้อมูลจะวิ่งไปรวบรวมที่ Sensor Gateway ในห้อง NECTEC

### ชั้น 4 — Dual Gateway (โครงข่ายเกตเวย์คู่)

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board จำนวน 1 ชุด ที่ห้อง Competitive และติดตั้ง Gateway จำนวน 2 ตัวที่ห้อง Hardware
- **การส่งข้อมูล**: ชั้นนี้ถูกออกแบบเพื่อความเสถียรของระบบ (Redundancy) โดยบอร์ดจะส่งข้อมูลผ่านโปรโตคอล **BLE / HTTP** ไปยังเกตเวย์ทั้งสองตัว เพื่อรับประกันว่าข้อมูลจะถูกส่งขึ้น Cloud ได้อย่างแน่นอนแม้มีเกตเวย์ตัวใดตัวหนึ่งขัดข้อง

## 3. สถาปัตยกรรมระบบคลาวด์ (Cloud Architecture & Data Processing)

โครงการนี้ขับเคลื่อนด้วยบริการบน **Microsoft Azure** โดยมีขั้นตอนการประมวลผลดังนี้:

- **Azure IoT Hub**: ทำหน้าที่เป็นจุดศูนย์กลางในการรับข้อมูลที่ถูกส่งมาจากอุปกรณ์ต่างๆ

- **Azure Stream Analytics**: รับสตรีมข้อมูลจาก IoT Hub และกระจายข้อมูลแบบเรียลไทม์ไปยัง 3 เส้นทางหลักเพื่อประมวลผลต่อ

- **การจัดเก็บและแสดงผล** (``Azure SQL DB → Web Application``): ข้อมูลจะถูกส่งไปบันทึกและจัดเก็บอย่างเป็นระบบที่ Azure SQL DB จากนั้นข้อมูลในฐานข้อมูลจะถูกดึงไปแสดงผลและใช้งานผ่าน Web Application เพื่อให้ผู้ใช้งานสามารถเข้าถึงได้

- **การวิเคราะห์ขั้นสูง** (``Azure Machine Learning → App Service``): ข้อมูลจะถูกส่งไปยัง Azure Machine Learning เพื่อนำไปวิเคราะห์ พยากรณ์ หรือสร้างโมเดลการเรียนรู้ ผลลัพธ์ที่ได้จะถูกนำไปให้บริการหรือแสดงผลผ่าน App Service

- **ระบบแจ้งเตือน** (``Event Hub → Azure Function → Email / Alert System``): ข้อมูลหรือเหตุการณ์ที่ต้องเฝ้าระวังจะถูกส่งไปยัง Event Hub เมื่อมีข้อมูลเข้ามา จะเกิดการกระตุ้น (Trigger) ให้ Azure Function ทำงาน โดย Azure Function จะประมวลผลและส่งการแจ้งเตือนไปยังผู้ใช้งานผ่านทาง Email โดยอัตโนมัติ นอกจากนี้เมื่อ Event Hub ตรวจพบความผิดปกติ จะส่งคำสั่งไปยัง Azure Function ซึ่งทำงานร่วมกับ Logic App และ Twilio เพื่อส่งข้อความแจ้งเตือนไปยังผู้ใช้งานโดยตรง

## 4. การสั่งการจากระยะไกล (Downlink Commands)

ผู้ดูแลระบบสามารถส่งคำสั่งจาก Cloud ลงมายังอุปกรณ์ (Downlink) ได้โดยตรง ประกอบด้วยคำสั่งหลัก:

- **สั่งถ่ายภาพ**: สั่งให้ ESP-CAM ภายในห้องบันทึกและส่งภาพกลับมา
- **สั่งรีเซ็ตระบบ**: สั่ง Reset Multimodal Board เพื่อเริ่มต้นการทำงานใหม่กรณีที่อุปกรณ์มีปัญหา

## 5. การออกแบบฐานข้อมูล (Database Design)

ข้อมูลสภาพแวดล้อมที่ถูกสตรีมมาจะถูกบันทึกลงใน **Azure SQL Database** โดยมีการออกแบบตารางหลักชื่อ `Telemetry` เพื่อรองรับการจัดเก็บข้อมูลแบบ Time-Series จากเซ็นเซอร์ต่างๆ ภายในอาคาร ดังโครงสร้างต่อไปนี้:

```sql
CREATE TABLE Telemetry (
    Id            BIGINT       IDENTITY(1,1) PRIMARY KEY, -- รองรับข้อมูลมหาศาล
    DeviceId      VARCHAR(50)  NOT NULL,                  -- รหัสอุปกรณ์ (ภาษาอังกฤษ/ตัวเลข)
    SensorType    VARCHAR(50)  NOT NULL,                  -- ประเภทเซ็นเซอร์
    SensorValue   FLOAT        NOT NULL,                  -- ค่าที่วัดได้
    QualityStatus TINYINT      DEFAULT 1,                 -- 1 = Good, 0 = Bad/Error
    CreatedAt     DATETIME2    DEFAULT GETUTCDATE()       -- บันทึกเวลา UTC เสมอ
);
```

### คำอธิบายโครงสร้างตาราง (Table Schema)

| คอลัมน์ | ชนิดข้อมูล | คำอธิบาย |
|---|---|---|
| `Id` | `BIGINT` | Primary Key แบบเพิ่มค่าอัตโนมัติ (IDENTITY) รองรับปริมาณข้อมูลมหาศาลจากเซ็นเซอร์อย่างต่อเนื่อง |
| `DeviceId` | `VARCHAR(50)` | รหัสประจำตัวของอุปกรณ์หรือโหนดที่ส่งข้อมูล ใช้ VARCHAR เพื่อประหยัดพื้นที่ เนื่องจากเป็นตัวอักษรภาษาอังกฤษหรือตัวเลข |
| `SensorType` | `VARCHAR(50)` | ประเภทของเซ็นเซอร์ที่ส่งค่ามา เช่น อุณหภูมิจาก DHT22, ระดับแสงจาก LDR Sensor หรือก๊าซจาก MQ-2 |
| `SensorValue` | `FLOAT` | ค่าตัวเลขที่วัดได้จากเซ็นเซอร์ รองรับข้อมูลที่เป็นจุดทศนิยม |
| `QualityStatus` | `TINYINT` | สถานะคุณภาพของข้อมูล: `1` = Good (ปกติ), `0` = Bad/Error (ผิดเพี้ยนหรือเซ็นเซอร์ขัดข้อง) |
| `CreatedAt` | `DATETIME2` | วันและเวลาที่บันทึก ตั้งค่า Default เป็น `GETUTCDATE()` เพื่อใช้มาตรฐาน UTC อำนวยความสะดวกแก่การวิเคราะห์เชิงพยากรณ์ (Predictive Analytics) |
