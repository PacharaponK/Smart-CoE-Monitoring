# โครงการ Smart CoE Monitoring

ระบบ IoT สำหรับตรวจวัดและรวบรวมข้อมูลสภาพแวดล้อมจากห้องต่างๆ ภายในอาคารภาควิชาวิศวกรรมคอมพิวเตอร์ (CoE) ข้อมูลทั้งหมดจะถูกส่งขึ้นระบบ Cloud เพื่อนำไปแสดงผล วิเคราะห์พยากรณ์ล่วงหน้า และแจ้งเตือนเมื่อเกิดความผิดปกติแบบเรียลไทม์

---

## 1. อุปกรณ์ฮาร์ดแวร์และเซ็นเซอร์ (Hardware & Sensor Nodes)

แต่ละห้องจะถูกติดตั้ง **Multimodal Board** ทำหน้าที่เป็นโหนดสำหรับวัดค่าสภาพแวดล้อมต่างๆ และส่งข้อมูลไปยัง Sensor Gateway โดยบอร์ดแต่ละตัวสามารถเลือกประกอบเซ็นเซอร์ได้ตามความเหมาะสมของแต่ละพื้นที่ใช้งาน ซึ่งรองรับเซ็นเซอร์ดังต่อไปนี้:

| เซ็นเซอร์ | หน้าที่ |
|---|---|
| **DHT11** | วัดอุณหภูมิและความชื้น |
| **LDR Sensor** | วัดความสว่างของแสง |
| **Sound Sensor** | ตรวจจับเสียง |

---

## 2. โครงสร้างเครือข่ายและเส้นทางการส่งข้อมูล (Network Topology by Floor)

การส่งข้อมูลถูกออกแบบให้เหมาะสมกับข้อจำกัดและลักษณะของแต่ละชั้นในอาคาร ดังนี้:

### ชั้น 2

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง R201, Co-Ai และ R200 พร้อมทั้งติดตั้งกล้อง ESP-CAM ที่ห้อง Co-CoE
- **การส่งข้อมูล**: อุปกรณ์เซ็นเซอร์จะส่งข้อมูลผ่านโปรโตคอล **ESP-NOW** หรือ **nRF Communication** มารวมกันที่ Sensor Gateway (ตั้งอยู่ที่ห้อง Co-CoE) จากนั้นจะส่งต่อไปยัง Network Gateway เพื่อเตรียมส่งขึ้นระบบ Cloud ต่อไป

### ชั้น 3 — High Availability Dual Gateway

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง **R303** และติดตั้ง Network Gateway จำนวน 2 ตัวที่ห้อง **PUPA** เพื่อทำระบบสำรอง (Redundancy)
- **การส่งข้อมูล**: โหนดเซ็นเซอร์ในห้อง R303 จะส่งข้อมูลผ่านโปรโตคอล **nRF Communication** ไปยังเกตเวย์ทั้งสองตัวพร้อมกัน หากเกตเวย์ตัวใดตัวหนึ่งขัดข้อง ข้อมูลจะยังคงถูกส่งขึ้นระบบ Cloud ได้อย่างต่อเนื่องผ่านเกตเวย์อีกตัวหนึ่ง ช่วยเพิ่มความเสถียรของระบบ (System Reliability) ในระดับสูงสุด

---

## 3. สถาปัตยกรรมระบบคลาวด์ (Cloud Architecture & Data Processing)

โครงการนี้ใช้สถาปัตยกรรม **AWS Serverless IoT** เต็มรูปแบบ เพื่อรองรับการขยายตัวในอนาคตและบริหารจัดการค่าใช้จ่ายตามการใช้งานจริง:

1. **การเชื่อมต่อ (Connectivity):** เกตเวย์เชื่อมต่อกับ **AWS IoT Core** ผ่านโปรโตคอล MQTT พร้อมการรักษาความปลอดภัยด้วย X.509 Certificates
2. **การประมวลผลข้อมูล (Data Orchestration):** **AWS IoT Rules Engine** ทำหน้าที่คัดแยกข้อมูลที่ได้รับและส่งต่อให้ **AWS Lambda** เพื่อประมวลผลตามเส้นทาง (Pipeline) ดังนี้:
    - **Data Ingestion Pipeline (สายที่ 1):** Lambda Function ทำหน้าที่ Parse ข้อมูลจากรูปแบบ Raw Data ตรวจสอบความถูกต้อง (Data Validation) และบันทึกลงใน **Amazon DynamoDB** ซึ่งถูกออกแบบโครงสร้างแบบ Time-series
    - **Real-time Alerting Pipeline (สายที่ 2):** เมื่อพบค่าที่สูงกว่าเกณฑ์ (Threshold) ที่ตั้งไว้ ระบบจะทริกเกอร์ Lambda เพื่อส่งการแจ้งเตือนเข้าสู่ **Discord Webhook** ทันที
3. **การบริการข้อมูลและการแสดงผล (Data API & Visualization):**
    - **API Layer:** ใช้ **AWS API Gateway** ทำหน้าที่เป็น Interface ให้กับ Web Application ในการ Query ข้อมูลจาก DynamoDB อย่างปลอดภัย
    - **Dashboard:** เว็บแอปพลิเคชัน (Next.js) จะดึงข้อมูลผ่าน API มาแสดงผลเป็นกราฟสถิติย้อนหลังและสถานะปัจจุบันแบบ Real-time

สถาปัตยกรรมนี้ตัดความยุ่งยากในการบริหารจัดการเซิร์ฟเวอร์ (Zero Infrastructure Management) และเน้นความยืดหยุ่นในการประมวลผลข้อมูลก่อนจัดเก็บเพื่อให้ได้ข้อมูลที่มีคุณภาพสูง

---

## 4. การสั่งการจากระยะไกล (Downlink Commands)

ผู้ดูแลระบบสามารถส่งคำสั่งจาก Cloud ลงมายังอุปกรณ์ (Downlink) ได้โดยตรง ประกอบด้วยคำสั่งหลัก:

- **สั่งถ่ายภาพ**: สั่งให้ ESP-CAM ภายในห้องบันทึกและส่งภาพกลับมา
- **ตั้งค่าเกณฑ์การแจ้งเตือน (Threshold Setting)**: ผู้ดูแลระบบสามารถกำหนดค่าเกณฑ์ (Threshold) ของเซ็นเซอร์แต่ละตัว เพื่อให้ระบบแจ้งเตือนเมื่อค่าที่วัดได้สูงเกินกำหนด (Trigger Discord Notification)

---

## 5. การออกแบบฐานข้อมูล (Database Design)

ข้อมูลสภาพแวดล้อมจะถูกจัดเก็บใน **Amazon DynamoDB** (NoSQL Database) ในรูปแบบ Time-Series โดยตาราง `Telemetry` จะถูกออกแบบมาเพื่อรองรับปริมาณข้อมูล (Throughput) จำนวนมหาศาลจากเซ็นเซอร์ด้วยโครงสร้างดังนี้:

### โครงสร้างตาราง (Table Schema)

| ส่วนประกอบ | ชื่อฟิลด์ | ชนิดข้อมูล | คำอธิบาย |
|---|---|---|---|
| **Partition Key** | `DeviceId` | `String` | รหัสประจำตัวของอุปกรณ์หรือโหนดที่ส่งข้อมูล (เช่น `COE-R201-01`) |
| **Sort Key** | `Timestamp` | `Number/String` | เวลาที่บันทึกข้อมูล (ISO8601 หรือ Epoch Time) เพื่อใช้ในการคัดกรองข้อมูลตามช่วงเวลา |
| **Attribute** | `SensorType` | `String` | ประเภทของเซ็นเซอร์ที่ส่งค่ามา เช่น `Temperature`, `Humidity`, `LDR` |
| **Attribute** | `SensorValue` | `Number` | ค่าตัวเลขที่วัดได้จากเซ็นเซอร์ (Float/Integer) |
| **Attribute** | `QualityStatus` | `Number` | สถานะคุณภาพของข้อมูล: `1` = Good (ปกติ), `0` = Bad/Error |

ด้วยโครงสร้างนี้ เราสามารถ Query ดึงข้อมูลย้อนหลังตามช่วงเวลาของแต่ละอุปกรณ์ (`DeviceId`) ได้อย่างรวดเร็วและมีค่าใช้จ่ายในการอ่าน/เขียนข้อมูลต่ำที่สุด โดยไม่ต้องกังวลเรื่องการจัดการ Scaling ของฐานข้อมูล
