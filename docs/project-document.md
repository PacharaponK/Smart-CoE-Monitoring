# โครงการ Smart CoE Monitoring

ระบบ IoT สำหรับตรวจวัดและรวบรวมข้อมูลสภาพแวดล้อมจากห้องต่างๆ ภายในอาคารภาควิชาวิศวกรรมคอมพิวเตอร์ (CoE) ข้อมูลทั้งหมดจะถูกส่งขึ้นระบบ Cloud เพื่อนำไปแสดงผล วิเคราะห์พยากรณ์ล่วงหน้า และแจ้งเตือนเมื่อเกิดความผิดปกติแบบเรียลไทม์

---

## 1. อุปกรณ์ฮาร์ดแวร์และเซ็นเซอร์ (Hardware & Sensor Nodes)

แต่ละห้องจะถูกติดตั้ง **Multimodal Board** ทำหน้าที่เป็นโหนดสำหรับวัดค่าสภาพแวดล้อมต่างๆ และส่งข้อมูลไปยัง Sensor Gateway โดยบอร์ดแต่ละตัวสามารถเลือกประกอบเซ็นเซอร์ได้ตามความเหมาะสมของแต่ละพื้นที่ใช้งาน ซึ่งรองรับเซ็นเซอร์ดังต่อไปนี้:

| เซ็นเซอร์ | หน้าที่ |
|---|---|
| **DHT22** | วัดอุณหภูมิและความชื้น |
| **IR Sensor** | อินฟราเรดตรวจจับวัตถุ |
| **LDR Sensor** | วัดความสว่างของแสง |
| **MQ-2** | ตรวจจับก๊าซไวไฟและควัน |
| **Sound Sensor** | ตรวจจับเสียง |
| **Hall Effect Sensor** | ตรวจจับสนามแม่เหล็ก |

## 2. โครงสร้างเครือข่ายและเส้นทางการส่งข้อมูล (Network Topology by Floor)

การส่งข้อมูลถูกออกแบบให้เหมาะสมกับข้อจำกัดและลักษณะของแต่ละชั้นในอาคาร ดังนี้:

### ชั้น 2

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง R201, Co-Ai และ R200 พร้อมทั้งติดตั้งกล้อง ESP-CAM ที่ห้อง Co-CoE
- **การส่งข้อมูล**: อุปกรณ์ทั้งหมดจะส่งข้อมูลผ่านโปรโตคอล **ESP NOW** มารวมกันที่ Sensor Gateway (ตั้งอยู่ที่ห้อง Co-CoE) จากนั้นจะส่งต่อไปยัง Network Gateway โดยเลือกใช้โปรโตคอลที่เหมาะสมตามระยะทาง เช่น BLE, ESP NOW หรือ RF Communication

### ชั้น 3

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board ที่ห้อง AI และห้อง Network
- **การส่งข้อมูล**: ข้อมูลของทั้งสองห้องจะถูกส่งผ่านโปรโตคอล **CoAP** ซึ่งมีน้ำหนักเบาและเหมาะสมอย่างยิ่งกับอุปกรณ์ที่มีทรัพยากรจำกัด โดยข้อมูลจะวิ่งไปรวบรวมที่ Sensor Gateway ในห้อง NECTEC

### ชั้น 4 — Dual Gateway (โครงข่ายเกตเวย์คู่)

- **จุดติดตั้ง**: ติดตั้ง Multimodal Board จำนวน 1 ชุด ที่ห้อง Competitive และติดตั้ง Gateway จำนวน 2 ตัวที่ห้อง Hardware
- **การส่งข้อมูล**: ชั้นนี้ถูกออกแบบเพื่อความเสถียรของระบบ (Redundancy) โดยบอร์ดจะส่งข้อมูลผ่านโปรโตคอล **BLE / HTTP** ไปยังเกตเวย์ทั้งสองตัว เพื่อรับประกันว่าข้อมูลจะถูกส่งขึ้น Cloud ได้อย่างแน่นอนแม้มีเกตเวย์ตัวใดตัวหนึ่งขัดข้อง

## 3. สถาปัตยกรรมระบบคลาวด์ (Cloud Architecture & Data Processing)

โครงการนี้ขับเคลื่อนด้วยสถาปัตยกรรม **AWS IoT แบบ Serverless** ที่เน้นความคุ้มค่าและความประหยัด (Cost-Optimization) โดยมีขั้นตอนการทำงานดังนี้:

1. **การเชื่อมต่อ (Connectivity):** อุปกรณ์ (Raspberry Pi/ESP32) เชื่อมต่อและส่งข้อมูลผ่านโปรโตคอล MQTT มายัง **AWS IoT Core** (Broker) อย่างปลอดภัย
2. **การคัดกรองข้อมูล (Rules Engine):** **AWS IoT Rules Engine** จะทำหน้าที่ตรวจสอบข้อมูลที่เข้ามา และแยกสายการทำงานเป็น 2 เส้นทางหลัก:
    - **สายที่ 1 (การบันทึกข้อมูลแบบประหยัด):** หากเป็นข้อมูลปกติ Rules Engine จะใช้ "Direct Action" เพื่อเขียนข้อมูลลงใน **Amazon DynamoDB** (NoSQL Time-series) โดยตรง *วิธีนี้ช่วยประหยัดค่าใช้จ่ายได้สูงสุดเนื่องจากไม่ต้องใช้ Lambda เป็นตัวกลาง*
    - **สายที่ 2 (การแจ้งเตือนตามเงื่อนไข):** หากข้อมูลตรงตามเงื่อนไขที่ผิดปกติ (เช่น อุณหภูมิสูงเกินกำหนด) Rules Engine จะทริกเกอร์เรียก **AWS Lambda** เพื่อประมวลผลโลจิกการแจ้งเตือน (Alert Logic) และสั่งให้ **Amazon SES (Email Dispatch)** ส่งอีเมลแจ้งเตือนไปยังผู้ดูแลระบบทันที
3. **การแสดงผล (Visualization):** ผู้ใช้งานสามารถเข้าดูแดชบอร์ดผ่าน Web Browser โดยระบบจะเชื่อมต่อเพื่อดึงข้อมูล (Query Data via HTTPS) จาก **Amazon DynamoDB** มาแสดงผลเป็นกราฟหรือค่าปัจจุบันแบบเรียลไทม์

สถาปัตยกรรมนี้ตัดบริการที่มีค่าใช้จ่ายสูง (เช่น Kinesis หรือ SageMaker) ออก และเปลี่ยนมาใช้ **DynamoDB + Direct Rule Writing** ซึ่งเป็น Best Practice สำหรับระบบ IoT ที่เน้นประสิทธิภาพและความประหยัดงบประมาณ

## 4. การสั่งการจากระยะไกล (Downlink Commands)

ผู้ดูแลระบบสามารถส่งคำสั่งจาก Cloud ลงมายังอุปกรณ์ (Downlink) ได้โดยตรง ประกอบด้วยคำสั่งหลัก:

- **สั่งถ่ายภาพ**: สั่งให้ ESP-CAM ภายในห้องบันทึกและส่งภาพกลับมา
- **สั่งรีเซ็ตระบบ**: สั่ง Reset Multimodal Board เพื่อเริ่มต้นการทำงานใหม่กรณีที่อุปกรณ์มีปัญหา

## 5. การออกแบบฐานข้อมูล (Database Design)

ข้อมูลสภาพแวดล้อมจะถูกจัดเก็บใน **Amazon DynamoDB** (NoSQL Database) ในรูปแบบ Time-Series โดยตาราง `Telemetry` จะถูกออกแบบมาเพื่อรองรับปริมาณข้อมูล (Throughput) จำนวนมหาศาลจากเซ็นเซอร์ด้วยโครงสร้างดังนี้:

### โครงสร้างตาราง (Table Schema)

| ส่วนประกอบ | ชื่อฟิลด์ | ชนิดข้อมูล | คำอธิบาย |
|---|---|---|---|
| **Partition Key** | `DeviceId` | `String` | รหัสประจำตัวของอุปกรณ์หรือโหนดที่ส่งข้อมูล (เช่น `COE-R201-01`) |
| **Sort Key** | `Timestamp` | `Number/String` | เวลาที่บันทึกข้อมูล (ISO8601 หรือ Epoch Time) เพื่อใช้ในการคัดกรองข้อมูลตามช่วงเวลา |
| **Attribute** | `SensorType` | `String` | ประเภทของเซ็นเซอร์ที่ส่งค่ามา เช่น `Temperature`, `Humidity`, `LDR` |
| **Attribute** | `SensorValue` | `Number` | ค่าตัวเลขที่วัดได้จากเซ็นเซอร์ (Float/Integer) |
| **Attribute** | `QualityStatus` | `Number` | สถานะคุณภาพของข้อมูล: `1` = Good (ปกติ), `0` = Bad/Error |

ด้วยโครงสร้างนี้ เราสามารถ Query ดึงข้อมูลย้อนหลังตามช่วงเวลาของแต่ละอุปกรณ์ (`DeviceId`) ได้อย่างรวดเร็วและมีค่าใช้จ่ายในการอ่าน/เขียนข้อมูลต่ำที่สุด โดยไม่ต้องกังวลเรื่องการจัดการ Scaling ของฐานข้อมูล
